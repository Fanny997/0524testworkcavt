# 自定义操作与工作流注册说明

本文说明如何用 **Nuclio-only** 的方式，把模型推理、数据增强或其他处理步骤注册到“工作流”页面。

当前约定：不再默认扫描 `local.*` 本地 Python handler。所有正式操作都包装成 Nuclio function，然后由线下目录自动部署和自动注册。

## 部署摘要

完整部署链路只有一条：

```text
准备 custom_operations_registry/<操作名>/manifest.json
  -> 准备 custom_operations_registry/<操作名>/nuclio/function.yaml 和 main.py
  -> 启动 Nuclio dashboard
  -> 执行 manage.py deploycustomoperations
  -> 后端部署 Nuclio function 并同步数据库
  -> 前端“工作流”页面自动显示“模型 / 数据增强”
```

常用命令：

```powershell
docker compose -f docker-compose.yml -f components/serverless/docker-compose.serverless.yml up -d
C:\ProgramData\anaconda3\envs\rgbtsar\python.exe manage.py deploycustomoperations
```

验证部署：

```powershell
nuctl get functions --platform local
C:\ProgramData\anaconda3\envs\rgbtsar\python.exe manage.py synccustomoperations
```

如果只想预览命令，不真正部署：

```powershell
C:\ProgramData\anaconda3\envs\rgbtsar\python.exe manage.py deploycustomoperations --dry-run
```

## Nuclio 基础镜像

正式新增模型时，不建议在每个 `function.yaml` 里写 `pip install onnxruntime`。那样每次部署新 function 都可能重新下载依赖，部署慢，也容易受网络影响。

推荐先构建一个统一的 Nuclio 基础镜像：

```powershell
docker build `
  -f custom_operations_registry/base-images/nuclio-onnxruntime/Dockerfile `
  -t cvat/nuclio-onnxruntime:dev `
  custom_operations_registry/base-images/nuclio-onnxruntime
```

这个镜像包含：

| 依赖 | 用途 |
| --- | --- |
| `onnxruntime` | 加载和执行 `.onnx` 模型。 |
| `opencv-python-headless` | 图像前处理、resize、颜色空间转换、后处理绘制等。 |
| `numpy` | 模型输入输出数组处理。 |
| `pillow` | 读取、保存和编码图片。 |

验证基础镜像：

```powershell
docker run --rm --entrypoint python cvat/nuclio-onnxruntime:dev -c "import onnxruntime, cv2, numpy, PIL; print('ok')"
```

后续新增模型或数据增强操作时，`function.yaml` 直接引用：

```yaml
spec:
  build:
    image: cvat.my-detector
    baseImage: cvat/nuclio-onnxruntime:dev
```

这样新增模型时只需要放 `main.py` 和 `models/*.onnx`，不会在每个 function 构建阶段重复下载 ONNX Runtime。

注意：当前示例的 `function.yaml` 已经依赖 `cvat/nuclio-onnxruntime:dev`。第一次运行 `python dev_start.py` 之前，请先完成上面的 `docker build`。否则 Nuclio 会尝试把它当作远程镜像拉取，导致部署失败。

## 目录约定

默认扫描目录固定为项目根目录下：

```text
custom_operations_registry/
```

每个子文件夹就是一个操作。**子文件夹名称默认就是页面显示名称和 `nuclio_function` 名称**。

推荐结构：

```text
custom_operations_registry/
  sample-onnx-detector/
    manifest.json
    nuclio/
      function.yaml
      main.py
      models/
        foreground_detector.onnx

  rgb-blur-augmentation/
    manifest.json
    nuclio/
      function.yaml
      main.py
```

如果 `manifest.json` 里没有写 `name`，默认使用子文件夹名，例如 `sample-onnx-detector`。

如果 `manifest.json` 里没有写 `nuclio_function`，默认也使用子文件夹名。这个值必须和 `nuclio/function.yaml` 里的 `metadata.name` 一致。

## 工作流程

1. 在 `custom_operations_registry/<operation-name>/` 下准备文件：
   - `manifest.json`
   - `nuclio/function.yaml`
   - `nuclio/main.py`
   - 可选模型文件，例如 `nuclio/models/foreground_detector.onnx`

2. 启动 Nuclio dashboard：

```powershell
docker compose -f docker-compose.yml -f components/serverless/docker-compose.serverless.yml up -d
```

3. 自动部署 Nuclio function 并同步到“工作流”：

```powershell
C:\ProgramData\anaconda3\envs\rgbtsar\python.exe manage.py deploycustomoperations
```

4. 打开“工作流”页面：
   - `kind = model` 的操作会出现在“模型”
   - `kind = augmentation` 的操作会出现在“数据增强”
   - 页面会从后端数据库读取 `input_schema`，自动生成文件上传框、参数输入框和前序步骤输出选择框。

只查看将要执行的部署命令：

```powershell
C:\ProgramData\anaconda3\envs\rgbtsar\python.exe manage.py deploycustomoperations --dry-run
```

只部署一个操作：

```powershell
C:\ProgramData\anaconda3\envs\rgbtsar\python.exe manage.py deploycustomoperations --only sample-onnx-detector
```

只同步数据库，不部署 Nuclio：

```powershell
C:\ProgramData\anaconda3\envs\rgbtsar\python.exe manage.py synccustomoperations
```

## manifest.json 顶层字段

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `name` | 否 | string | 页面显示名称。未写时使用子文件夹名称。 |
| `kind` | 是 | string | 操作类型：`model` 或 `augmentation`。模型和数据增强都可以包装成 Nuclio function。 |
| `description` | 否 | string | 页面说明文本。 |
| `nuclio_function` | 否 | string | Nuclio function 名。未写时使用子文件夹名称。必须和 `function.yaml` 的 `metadata.name` 一致。 |
| `artifact` | model 通常需要 | string | 模型或资源文件路径。相对路径基于 manifest 所在目录。 |
| `artifact_builder` | 否 | string | 用 Python 函数生成 artifact 的导入路径。Nuclio-only 模式一般不需要。 |
| `artifact_name` | 否 | string | 使用 `artifact_builder` 时的 artifact 文件名。 |
| `input_schema` | 是 | array | 输入字段定义。支持 N 个输入集合。 |
| `output_schema` | 否 | object | 输出结构说明，不强制校验返回值。 |
| `nuclio` | 是 | object | Nuclio 自动部署配置。 |
| `is_active` | 否 | boolean | 是否启用。默认 `true`。 |

不要在 Nuclio-only 模式里写：

```json
{
  "nuclio_function": "local.xxx",
  "handler": "some.python:function"
}
```

`local.*` 和 `handler` 已不作为默认扫描方式使用。

### 前端如何知道有哪些操作

前端不直接扫描文件夹，也不直接访问 Nuclio。流程是：

```text
manifest.json
  -> synccustomoperations 写入 CustomOperationDefinition 数据库表
  -> “工作流”页面调用 /api/custom-operations/definitions?kind=model 或 kind=augmentation
  -> 前端按 input_schema 渲染操作所需输入
```

因此：

- `kind = model` 决定它出现在“模型”列表。
- `kind = augmentation` 决定它出现在“数据增强”列表。
- `input_schema` 决定页面显示哪些输入集合、参数、类型限制。
- `accept` 决定上传文件类型检查，也决定从前序步骤输出里选哪个文件。
- `output_schema` 只做说明，不负责渲染输入。

### 模型 artifact 是否必须

Nuclio-only 模式下，模型文件有两种放法：

1. 放在 Nuclio function 镜像或 function 目录里，由 `main.py` 自己读取。
2. 写在 manifest 的 `artifact` 字段里，由 CVAT 同步到 media 目录，并通过 `payload["cvat_context"]["artifact_url"]` 提供给 Nuclio。

所以 manifest 自动同步允许 `kind = model` 但没有 `artifact`。如果你的 Nuclio function 已经把模型打包进去了，就可以不写 `artifact`。

手动 API 注册仍然会要求 model 上传 artifact，因为手动注册入口没有 Nuclio 部署上下文。

## nuclio 字段

`nuclio` 字段告诉 `deploycustomoperations` 怎么调用 `nuctl deploy`。

| 字段 | 必填 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `path` | 否 | string | `nuclio` | Nuclio function 目录。相对路径基于 manifest 所在目录。 |
| `file` | 否 | string | `<path>/function.yaml` | Nuclio 配置文件。 |
| `project_name` | 否 | string | `cvat` | Nuclio project 名。 |
| `platform` | 否 | string | `local` | 本地 Docker 部署使用 `local`。 |
| `network` | 否 | string | 无 | Docker 网络名。本项目 compose 通常使用 `cvat_cvat`。 |
| `platform_config` | 否 | object | 根据 `network` 自动生成 | 直接传给 `nuctl deploy --platform-config`。 |
| `create_project` | 否 | boolean | `true` | 部署前是否执行 `nuctl create project`。 |
| `offline` | 否 | boolean | `true` | 部署时追加 `--offline`，避免 Nuclio 访问远程镜像仓库。 |
| `no_pull` | 否 | boolean | `true` | 部署时追加 `--no-pull`，优先使用本地已有基础镜像和 Nuclio 辅助镜像。 |
| `env` | 否 | object | `{}` | 环境变量，会展开成 `--env KEY=VALUE`。 |
| `deploy_args` | 否 | string[] | `[]` | 额外追加到 `nuctl deploy` 的参数。 |

最常用配置：

```json
"nuclio": {
  "path": "nuclio",
  "file": "nuclio/function.yaml",
  "project_name": "cvat",
  "platform": "local",
  "network": "cvat_cvat",
  "offline": true,
  "no_pull": true,
  "create_project": true
}
```

## input_schema 字段

`input_schema` 是数组。每个元素定义一个输入控件或参数。

通用字段：

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `name` | 是 | string | 字段名，也是 Nuclio payload 里的 key。必须唯一。 |
| `label` | 否 | string | 页面显示名。 |
| `type` | 是 | string | 输入类型。 |
| `required` | 否 | boolean | 是否必填。 |
| `description` | 否 | string | 表单说明。 |
| `placeholder` | 否 | string | 占位提示。 |
| `default` | 否 | any | 默认值。 |

支持的 `type`：

| type | 用途 |
| --- | --- |
| `string` | 单行文本。 |
| `text` | 多行文本。 |
| `integer` | 整数。 |
| `number` | 浮点数。 |
| `boolean` | 开关。 |
| `select` | 下拉选择。 |
| `json` | JSON 参数。 |
| `file` | 单文件上传，保留兼容。 |
| `file_collection` | 文件集合，推荐用于批量图片、txt、标注文件。 |

文件集合字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `accept` | string[] | 文件类型白名单，例如 `image/png`、`image/*`、`.txt`。 |
| `min_count` | integer | 最少文件数，默认 1。 |
| `max_count` | integer | 最多文件数。 |

数值字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `minimum` | number | 最小值。 |
| `maximum` | number | 最大值。 |
| `step` | number | 步长。 |

下拉字段：

```json
{
  "name": "operation",
  "label": "操作",
  "type": "select",
  "default": "blur",
  "options": [
    { "label": "模糊", "value": "blur" },
    { "label": "反色", "value": "invert" }
  ]
}
```

## N 个输入集合

一个操作可以定义 N 个 `file_collection` 输入集合。

如果只有 1 个集合，上传 N 个文件就执行 N 次。

如果有多个集合，所有集合数量必须一致，按上传顺序配对：

```text
image[0] + txt[0] + mask[0]
image[1] + txt[1] + mask[1]
...
```

示例：图片 + txt 两个输入集合：

```json
[
  {
    "name": "image",
    "label": "Images",
    "type": "file_collection",
    "required": true,
    "accept": ["image/png", "image/jpeg"]
  },
  {
    "name": "meta",
    "label": "Metadata files",
    "type": "file_collection",
    "required": true,
    "accept": ["text/plain", ".txt"]
  }
]
```

上传 10 张图片和 10 个 txt 时，后端执行 10 次。数量不一致会报错。

## 前序步骤输出作为输入

“工作流”支持在第 2 步及之后，为每个 `file_collection` 输入选择来源：

- “上传文件”：重新上传文件。
- “使用第 1 步输出”
- “使用第 2 步输出”
- ...

也就是说，当前操作的每个输入集合都可以独立选择任意前序步骤的输出。

例如第 3 步有两个输入集合：

```text
image
meta
```

可以选择：

```text
image <- 使用第 1 步输出
meta  <- 上传文件
```

或者：

```text
image <- 使用第 1 步输出
meta  <- 使用第 2 步输出
```

后端会把被选中的前序步骤输出当作一个输入集合。即使上一步每个 item 里有多个文件，下游也只看到一个集合；后端会根据当前输入字段的 `accept` 从每个 item 中选择匹配文件。

## 输出集合

无论一个操作内部输出多少文件，“工作流”都把它归一化为一个 `output_collection`：

```json
{
  "run_id": 12,
  "count": 10,
  "items": [
    {
      "index": 0,
      "files": [
        {
          "name": "01_yolo_0001__cat.json",
          "path": "C:\\tmp\\outputs\\01_yolo_0001__cat.json",
          "content_type": "application/json",
          "kind": "text"
        }
      ],
      "result": {}
    }
  ]
}
```

如果 Nuclio 函数返回文件路径，后端会把这些文件放入集合。若只返回 JSON 结果，后端会自动保存为 `.json` 文件并放入集合。

## Nuclio payload

批量输入会在 CVAT 后端拆开。Nuclio 每次收到当前配对后的单个 item。

图片字段示例：

```python
import base64
import io
from PIL import Image

image_payload = data["image"]
raw = base64.b64decode(image_payload["data"])
image = Image.open(io.BytesIO(raw)).convert("RGB")
```

artifact 信息：

```python
artifact_url = data["cvat_context"]["artifact_url"]
artifact_name = data["cvat_context"]["artifact_name"]
```

输出目录和建议文件名前缀：

```python
output_dir = data["cvat_context"]["output_dir"]
output_prefix = data["cvat_context"]["output_prefix"]
```

批处理信息：

```python
batch = data["cvat_context"]["batch"]
index = batch["index"]
count = batch["count"]
```

## Nuclio 返回格式

返回单个输出文件：

```json
{
  "output": {
    "name": "result.png",
    "path": "C:\\tmp\\outputs\\result.png",
    "content_type": "image/png",
    "kind": "image"
  }
}
```

返回多个输出文件：

```json
{
  "outputs": [
    {
      "name": "result.png",
      "path": "C:\\tmp\\outputs\\result.png",
      "content_type": "image/png",
      "kind": "image"
    },
    {
      "name": "result.txt",
      "path": "C:\\tmp\\outputs\\result.txt",
      "content_type": "text/plain",
      "kind": "text"
    }
  ]
}
```

只返回 JSON：

```json
{
  "type": "object_detection",
  "detections": []
}
```

只返回 JSON 时，后端会自动保存 JSON 文件。

## function.yaml 最小模板

`metadata.name` 必须等于操作子文件夹名，除非 manifest 显式写了 `nuclio_function`。
`deploycustomoperations` 会在部署前校验这两个值，不一致会直接报错，避免“工作流”页面能看到操作但后端调用不到 Nuclio function。

```yaml
metadata:
  name: my-yolo-detector
  namespace: cvat

spec:
  description: 自定义操作
  runtime: python:3.10
  handler: main:handler
  eventTimeout: 60s

  build:
    image: cvat.my-yolo-detector
    baseImage: cvat/nuclio-onnxruntime:dev
    directives:
      preCopy:
        - kind: USER
          value: root
        - kind: WORKDIR
          value: /opt/nuclio
        - kind: ENTRYPOINT
          value: '["/usr/local/bin/processor"]'

  triggers:
    myHttpTrigger:
      kind: http
      numWorkers: 2
      attributes:
        maxRequestBodySize: 33554432

  platform:
    attributes:
      restartPolicy:
        name: always
        maximumRetryCount: 3
      mountMode: volume
```

## main.py 最小模板

```python
import base64
import io
import json

from PIL import Image


def handler(context, event):
    data = event.body
    image_payload = data["image"]
    raw = base64.b64decode(image_payload["data"])
    image = Image.open(io.BytesIO(raw)).convert("RGB")

    result = {
        "type": "example",
        "input": {
            "name": image_payload["name"],
            "width": image.width,
            "height": image.height
        }
    }

    return context.Response(
        body=json.dumps(result),
        content_type="application/json",
        status_code=200,
    )
```
